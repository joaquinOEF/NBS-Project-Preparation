Below is a **single, structured table** you can use as the **decision backbone** when a user selects an asset inside a zone. It ties together:

* **Intervention category**
* **OSM asset type**
* **Concrete intervention actions**
* **Typical scale**
* **Order-of-magnitude costs**
* **Expected impacts (by hazard)**
* **Key implementation considerations**

This is intentionally **practical, not academic**. Values are indicative ranges suitable for comparison and scenario-building, not procurement.

## How this table is used in the app (important)

When a user double-clicks a zone:

1. **Zone typology filters** which rows are relevant
2. **OSM asset type filters** which interventions are possible
3. The table provides:

   * a *menu* of interventions
   * a *default assumption* for cost and impact
   * contextual notes for decision-making

You are **not forcing a single solution** — you are constraining choices to those that make sense.

---

## How to frame “impact” in the MVP (don’t overpromise)

Use **relative impacts**, not absolute predictions:

* Flood: ↓ peak accumulation / ↓ pooling likelihood (low / medium / high)
* Heat: ↓ exposure to extreme heat (especially population-weighted)
* Landslide: ↓ triggering probability

Later, these can be turned into:

* % score reduction per unit area
* scenario deltas (“with / without intervention”)

## Why this works well for decision support

* Keeps decisions grounded in **real places**
* Makes trade-offs visible (cost vs benefit)
* Supports **multi-hazard reasoning**
* Scales across cities without custom tuning
